# runbook.md — operator-steps artifact

A first-class feature artifact for the steps that aren't code: provisioning, secrets, console-only configuration, manual DNS, deploy-day toggles. Generated by `engineer:plan` when the plan touches infrastructure. Lives at `features/NNN-<slug>/runbook.md`.

**Why this exists.** image-titler's 016-staging-environment feature spent ~100 conversation events on Firebase staging bring-up (link billing, enable 11 APIs, create Firestore + Storage + 2 hosting sites + web app + service account + GitHub secrets) — entirely outside the DAE artifact model. The agent wrote ad-hoc Phase-0 runbooks in `progress.md` and had to ask the human for console-only steps twice. Acceptance tests claimed green while the infrastructure they depended on wasn't fully provisioned. `runbook.md` makes operator steps visible, ownership-tagged, and gate-enforced.

## Schema

```yaml
---
slug: 016-staging-environment
checkpoint: 4                              # generated as part of the plan
created: 2026-06-04
status: open                               # open | partial | complete
steps:
  - id: gcp-project-create
    description: "Create GCP project 'image-titler-staging'"
    owner: human                           # human | agent
    command: null                          # null when owner: human and step is console-only
    evidence: null                         # filled when complete: URL, screenshot path, or output
    completed: false
    blocking_acs:                          # ACs that cannot claim green until this step completes
      - AC-deploy-staging-url

  - id: enable-firestore-api
    description: "Enable Firestore API in the GCP project"
    owner: agent
    command: "gcloud services enable firestore.googleapis.com --project=image-titler-staging"
    evidence: null
    completed: false
    blocking_acs:
      - AC-deploy-staging-url

  - id: github-secret-firebase-token
    description: "Add FIREBASE_TOKEN to GitHub Actions secrets"
    owner: human
    command: null
    evidence: null
    completed: false
    blocking_acs:
      - AC-ci-deploys-on-push
---

# Runbook — image-titler-staging

## Phase 0 — Provision (one-time)

Section bodies are human-readable narrative — what to do, why, gotchas. The
steps frontmatter is the machine-readable source of truth for the gate logic.

…
```

## Required fields per step

- `id` — kebab-case, unique within the runbook.
- `description` — one line, imperative ("Create…", "Enable…", "Add…").
- `owner` — `human` (console-only, decision-required, or secrets) or `agent` (runnable command, no human-in-the-loop).
- `command` — the literal command, or `null` if `owner: human`.
- `evidence` — URL / output / "confirmed in console" / etc. `null` until `completed: true`.
- `completed` — boolean.
- `blocking_acs` — list of AC IDs (from `acs.md`) that depend on this step. Empty list `[]` for steps that block nothing (rare; if a step blocks nothing, it probably doesn't belong in the runbook).

## Gate enforcement

Deploy-related ACs MUST NOT claim green until every step in their `blocking_acs` reverse-lookup is `completed: true`. `engineer:verify` will surface this as a gap; `dae_handoff.py` will refuse to mark the CP7 checkpoint complete if any blocked AC is asserting `met: true` while its blocking runbook steps are open.

Use `dae_runbook.py status <feature-dir>` (if available; otherwise grep) to see what's open vs done.

## Status lifecycle

- `open` — at least one step incomplete.
- `partial` — some steps complete, but acceptance ACs that depend on the open ones are not yet green.
- `complete` — every step `completed: true`. `runbook.md` becomes a frozen historical record at this point.

## When to write a runbook

`engineer:plan` should write one when the plan's Architecture section mentions any of:
- Cloud-provider provisioning (GCP/AWS/Azure project, IAM, billing).
- Secrets generation, storage, or rotation (service-account keys, API tokens).
- DNS, TLS certificates, CDN configuration.
- Console-only feature flags or toggles.
- One-time data migrations or seeding.
- External service connection (Stripe, SendGrid, Sentry — anything that requires a UI step on a third-party).

If the plan is pure code change (refactor, new endpoint on an existing service, UI on an existing app), no runbook — don't write an empty file.
