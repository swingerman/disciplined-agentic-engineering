From a plugin to a methodology: Disciplined Agentic Engineering

A few weeks ago I packaged Robert C. Martin's acceptance-test approach as a Claude Code plugin. That was the seed. Watching where Uncle Bob took it next is what grew it into a whole methodology.

He kept going — running a swarm of Claude agents for days at a stretch. And the swarm started to fail in instructive ways:

"They've started to lose their particular identities. The coder is doing crap analysis. The refactorer finds it has nothing to do because the coder did it... The architect decided to not run mutation tests because they take a long time."
None of that is the model being dumb. It is a methodology gap. Discipline that lives in a prompt is soft — the agent rationalizes past it, and it erodes every time the context compacts. Uncle Bob put the fix precisely:

"You, the architect, must drive that vision by constructing the guard rails that enforce it. Those guard rails are not AI rules — they are independent tools that the AI uses to validate what it has built."
So I built Disciplined Agentic Engineering (DAE). One sentence: humans direct, agents type. Engineers own architecture, behavior decisions, and verification; agents do the keystrokes. Discipline lives in contracts at every layer — not in good intentions.

WHAT THAT LOOKS LIKE

  • A charter — the binding design agreement every checkpoint enforces.
  • An 8-checkpoint pipeline per feature — each checkpoint with a verifiable exit contract, not a vibe. A stage is done only when a tool says so.
  • Verification independence — the agent that wrote the code is never the one that signs it off.
  • Context resilience — prefer fresh, checkpoint-scoped runs over marathon agents; re-anchor after every compaction.
  • A charter architecture fitness tool — your layering and convention rules, checked like a test.
  • Test impact analysis — change a file, run only the scenarios it actually affects.
Every one of those started as a failure Uncle Bob hit in public, and became a guard rail — an independent tool, not a rule in a prompt.

THE KIT

DAE ships as three Claude Code plugins:

  • engineer — the DAE pipeline: feature intake, AC discovery, spec writing, planning, refinement, verification.
  • atdd — Uncle Bob's acceptance pipeline: standard Gherkin specs, a project-specific test generator, two test streams.
  • crap-analyzer — change-risk analysis on the code you just touched.
Install it: 

/plugin marketplace add swingerman/disciplined-agentic-engineering

Or read the methodology: github.com/swingerman/disciplined-agentic-engineering

The throughline is the one Uncle Bob keeps landing on: the newer things are, the more relevant the old rules become. AI did not retire software-engineering discipline. It made it the bottleneck — and the thing most worth automating.

Full credit to Robert C. Martin. The experiments, the struggles, and the lessons are his. DAE is an attempt to make that discipline the path of least resistance.
