The discipline layer for the spec-as-source-code era

Three months in, Disciplined Agentic Engineering (DAE) hit v1.3 — a Claude Code methodology for engineers who want AI's speed without losing the system's coherence.

The premise. Vibe coding is the rising practice of prompting an AI loosely — "build me a thing that does X" — accepting whatever comes back, patching when it feels off. Fast at first; brittle over time. The codebase drifts; tests pass without proving anything; regressions surface as users find them.

DAE is the deliberate opposite. Every feature has a charter to obey, a Ready contract, ACs in domain language, a Gherkin spec the human approves, an architecture plan, two test streams that both must go green, mutation testing as the test-quality firewall — and deterministic guardrail tools gating every transition.

Discipline lives in tools, not prompts. Discipline that lives in a prompt erodes; discipline that lives in independent scripts the agent has to run and pass doesn't. Every checkpoint in DAE's pipeline is gated by a tool:

  • dae_handoff.py — handoff-as-gate: a checkpoint isn't done until its handoff exists with every exit criterion met
  • dae_branch.py — branch hygiene at every checkpoint
  • dae_arch.py — charter architecture fitness (layering, cycles, naming, file size)
  • dae_dup.py — project-wide duplicate detection fed into Refine's Reuse lens
  • dae_mutmap.py — differential mutation testing (re-mutate only what changed)
  • dae_impact.py — test impact analysis on the acceptance pipeline
  • dae_progress.py — pipeline breadcrumb at every Step 0
Plus skill instructions for LSP-first code lookup, four-pass AC discovery, charter-bound refinement, fresh-per-phase agent teams, and structured handoffs that the next checkpoint's entry gate verifies.

Where this fits in software's history. Programming has always climbed the abstraction ladder — binary → assembly → high-level → frameworks → and now, with capable LLMs, spec and behavior description as the next rung. Nobody hand-writes opcodes anymore; nobody audits the assembly a compiler emits. The rung below is trusted because the generator is rigorous AND the inputs are checked. AI-generated code earns that same trust the same way — rigorous inputs (charter, ACs, specs, plans), checked outputs (two test streams + mutation testing + arch fitness + change risk + duplicate detection + cycle detection).

What never shifts is engineering discipline. The methodology adapts; the engineering values are constant.

Who it's for. Software engineers who know what acceptance tests are, why mutation testing matters, why architectural layering matters. Not for "build an app without code" audiences — the decisions DAE puts in front of you are engineering decisions and need engineering judgment.

Three Claude Code plugins:

  • engineer — the DAE pipeline (16 skills, 10 guardrail scripts, 214 stdlib-only tests)
  • atdd — acceptance test driven workflow with differential mutation testing
  • crap-analyzer — change-risk analysis on the code you just touched
Install: 
/plugin marketplace add swingerman/disciplined-agentic-engineering
Or read the methodology: github.com/swingerman/disciplined-agentic-engineering

ATDD itself is an older testing practice from the XP / FIT / Fitnesse lineage (Kent Beck, Ward Cunningham, and others). Robert C. Martin's recent experiments running agent swarms on empire-2025 surfaced its application as a discipline against AI agentic drift — the immediate spark for the atdd plugin and the broader DAE methodology.
